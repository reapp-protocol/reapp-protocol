# Security Audit — MandateRegistry (testnet milestone)

- **Date:** 2026-06-10
- **Target:** `contracts/mandate-registry` (soroban-sdk 22)
- **Method:** 12-agent adversarial sweep — 6 attack surfaces, **each finding independently re-verified against the code**, then a synthesis verdict.
- **Verdict:** **airtight-ship** — 5 candidate findings, **0 confirmed defects**.

## Surfaces reviewed
Arithmetic & overflow · Authorization (`require_auth`) · Replay & sequencing ·
Token interaction & reentrancy · State machine & storage · Logic / economic correctness.

## Why it holds (auditor-confirmed)
- `require_auth` binds to the **stored** agent, not a caller-supplied address.
- State is persisted (`set_mandate`) **before** the external `transfer_from` — checks-effects-interactions ordering, so **no reentrancy window**.
- The SEP-41 allowance is an **independent hard ceiling** beneath the contract's own budget check (proven by `insufficient_allowance_blocks_payment`).
- `overflow-checks = true` in the release profile → `spent += amount` / `seq += 1` **panic-revert, never wrap**; within validated bounds overflow is unreachable anyway.
- `register_mandate` forces `spent=0 / seq=0 / status=Active`, so a caller cannot seed tampered state.
- Mandate-layer replay guard (`expected_seq == seq`, else `BadSequence`) sits on top of Soroban's transport nonce.

## Fixed during this pass
- Implemented mandate-layer replay enforcement (`seq` / `BadSequence`) — was previously dead.
- Removed dead `NotAuthorized` (auth is host-enforced via `require_auth`).
- Expiry symmetry: `check` rejects `now >= expiry` to match `register`'s `expiry > now`.
- `cargo fmt` clean, `cargo clippy` 0 warnings.

## Deferred to mainnet hardening (NOT testnet blockers)
1. **Asset allowlist** — a user may register against an arbitrary token; blast radius is self-inflicted, but add a SAC/allowlist guard before mainnet.
2. **Storage TTL vs `mandate.expiry`** — `get_mandate` does not bump TTL; a registered-but-never-paid mandate could expire from persistent storage after the 30-day TTL even if its own `expiry` is later. Bump TTL on read or align the entry TTL to `mandate.expiry`.
3. **`validate_and_consume` naming** — a read-only dry-run; the name is kept because the Tranche-1 deliverable lists it explicitly, and it is documented as a preflight. Revisit before the public SDK if desired.
