# Historical security gate check: legacy x402 surface (2026-06-16)

> Point-in-time record for proof-v1 before bound-v2. It is not evidence for
> core 0.3.0 or Express middleware 0.2.1. The requester/resource binding and
> persistent replay findings below are addressed by current bound-v2 code and
> tests; current evidence is recorded in `docs/T2-SUBMISSION.md`.

- **Date:** 2026-06-16
- **Target:** `packages/sdk/src/x402.ts`, `Agent.fetch` in `packages/sdk/src/index.ts`, `apps/fulfillment-agent/src/server.ts`, `apps/consumer-agent/src/research-agent.ts`, and `scripts/e2e-x402.ts`, against the testnet contract payment path (`payment.rs`, `events.rs`).
- **Method:** multi-agent adversarial review adversarial sweep. 23 agents across 6 attack surfaces. Every candidate finding was independently re-verified against the source by a separate skeptic, several reproduced empirically, then a completeness critic looked for missed paths.
- **Recorded result:** the review found one **critical** access-control bypass and one **medium** replay window; both were fixed in that historical version and the critical fix was verified on-chain.

## The bar

The merchant is the access-control boundary for the resource; it must never serve without independently confirming, on-chain, a real payment to itself. The SDK is untrusted; `Agent.fetch` must never bypass the contract (settlement always goes through `execute_payment`, the 402 is only a hint). The x402 wire format must be decoupled from the mandate and contract logic.

## Fixed during this pass

1. **Critical: merchant honored a forged `payment` event (access-control bypass).** `verifyOnChain` matched the event topic (`payment`, merchant) and amount but never checked which contract emitted it. Any contract could publish a forged `("payment", merchant, price)` event; the attacker presents that transaction as the `X-PAYMENT` proof and unlocks the resource with no real payment. **Fix:** the merchant now requires the event to be emitted by the MandateRegistry (`StrKey.encodeContract(ev.contractId()) === TESTNET.mandateRegistryId`). Verified on-chain against a real payment: the token's SEP-41 `transfer` event is correctly ignored, and only the registry's `payment` event is honored.
2. **Medium: replay TOCTOU.** The "already redeemed" check ran before the awaited on-chain verification, so two concurrent requests reusing one proof could both pass during the await window. **Fix:** the proof is reserved synchronously before the await and released only on a verification failure, so a transient RPC lag does not burn a real payment.
3. **Low: brittle e2e gate.** The end-to-end pass condition hardcoded friendbot's 10,000 XLM starting balance. **Fix:** it now asserts the merchant balance increased by exactly 3 XLM (a delta), robust to whatever friendbot funds.

## What held after the recorded fixes

- **`Agent.fetch` cannot bypass the contract.** Settlement always routes `fetch` to `pay` to `execute_payment`; the 402 body is never treated as authorization. The contract reads merchant and asset from stored mandate state, so a forged 402 cannot redirect the payment, and a revoked, expired, over-budget, or out-of-scope request is rejected on-chain (`fetch` throws).
- **The merchant verification is sound (post-fix).** It confirms the transaction succeeded, the `payment` event came from the MandateRegistry and paid this merchant at least the price, and the proof was not redeemed before. Hostile `X-PAYMENT` shapes (bad base64, non-JSON, wrong types, missing hash) all fail closed to a 402.
- **The x402 wire format is decoupled.** All HTTP shape lives in `x402.ts`; the contract and mandate import nothing from it. The merchant consumes only the proof's transaction hash and re-derives merchant and amount from the chain, ignoring every other client-supplied field.
- **The ResearchAgent is an exemplary safe pattern.** It holds no budget, settles every purchase through the contract, pays sequentially (matching the on-chain sequence guard), and surfaces rejections rather than retrying them.

## Deferred to mainnet hardening (not testnet blockers)

- **Per-call price ceiling.** `fetch` pays whatever amount the (untrusted) 402 demands, capped only by the mandate's total budget. The contract bounds total spend and the user chose the merchant, but a per-call cap would limit a compromised merchant to one over-charge.
- **Bind the unlock to the requester, the resource, and recency; persist the dedup (the main mainnet item).** Today the merchant honors any successful MandateRegistry payment to itself of at least the price, deduped by an in-process set. With a uniform price and a single-process demo this gives one-payment-one-unlock, but for a public or multi-instance deployment: bind the proof to the mandate or agent (so merely observing a public transaction hash is not enough to redeem it), reject payments older than a short ledger window, and persist the spent-transaction set so a restart cannot re-open past payments. The follow-up gatecheck flagged this as the one item to close before a public or multi-instance mainnet deployment; it is not a testnet blocker for the single-process reference demo.
- **Bind a payment to a specific resource.** The merchant matches "a real payment of at least the price to me", not "the payment for this exact request". With a uniform price this is economically equivalent, but per-resource binding (carry and check the resource id) is needed before variable pricing.
- **Decimals coupling.** The merchant's price threshold is a fixed 7-decimal stroop value. For native XLM this always agrees with the SDK; derive it from the asset's on-chain decimals before supporting other assets.
- **Structured error classification.** The reference agent classifies contract rejections by substring (`#6`); enforcement does not depend on it (it is display-only), but branch on the typed `Errors` code before mainnet. (`decodePaymentProof` already validates the proof shape strictly.)
- **Cross-check the SEP-41 transfer event.** The merchant trusts the contract's `payment` event, which is emitted strictly after a real `transfer_from` in the same atomic transaction, so it implies funds moved. Also confirming the token's `transfer` event would defend against a future contract change.

## Reproduce

```
npm run build
npm run e2e:x402
```

The end-to-end run settles three real payments through the merchant's on-chain verification and shows the fourth rejected by the contract.
