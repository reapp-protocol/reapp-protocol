# Security Gate check: REAPP SDK (`@reapp-sdk/core` + `@reapp-sdk/stellar`)

- **Date:** 2026-06-15
- **Target:** `packages/sdk` (`@reapp-sdk/core`, gate-checked at 0.1.2) and `packages/stellar` (`@reapp-sdk/stellar`, gate-checked at 0.1.1), plus the published `dist/` artifacts and `package.json` manifests.
- **Method:** BulletproofBar adversarial sweep. 31 agents across 8 attack surfaces. Every candidate finding was independently re-verified against the actual source by a separate skeptic agent (refute-by-default), then a completeness critic looked for surfaces and edges the sweep missed. Several findings were reproduced empirically against the installed `@stellar/stellar-sdk`.
- **Verdict:** **airtight for testnet.** 22 candidate findings, **0 confirmed defects**, 0 testnet blockers. Two real low-severity input-bound gaps were fixed in `@reapp-sdk/core` 0.1.2 during this pass. The rest are info-level confirmations or pre-mainnet hygiene.

## The bar

The SDK is **untrusted infrastructure**. The Soroban MandateRegistry contract is the only enforcement layer and the source of truth. A finding only counts as a defect if it can do one of these from the SDK side: move a wrong on-chain value, leak custody, bypass the contract's enforcement, leak a secret, or compromise a published package. Footguns and doc mismatches are low or info, not high. The milestone is testnet, so testnet blockers are separated from mainnet hardening.

## Surfaces reviewed

Amount parsing and money math · Custody boundary and allowance · SDK-cannot-bypass-the-contract · Mandate-id canonicalization · Network config and contract-id integrity · Secret and signer handling · Error surfacing and failure modes · Supply-chain and package hygiene.

## What holds (gate-check-confirmed)

- **The SDK has exactly one money path.** `Agent.pay` is the only spend, and it calls the contract's `execute_payment`. The SDK exposes no `transfer` / `transfer_from`; the real transfer fires only inside the contract, gated by `require_auth(agent)`. There is no alternate path to move funds.
- **No trusted local limit.** `pay` never checks the amount against a local budget. It re-reads the current sequence from chain on every call (no cached state) and passes everything to the contract, which re-validates seq, status, expiry, merchant scope, and budget atomically. A malicious or buggy SDK that sent an oversized amount or a stale sequence is still rejected on-chain (`BudgetExceeded`, `BadSequence`).
- **Custody never touches the SDK.** `approveBudget` grants the SEP-41 allowance to the contract (`mandateRegistryId`) as spender, never to the agent or SDK. The user signs that approval. The SDK holds no allowance and cannot pull funds.
- **The recipient is not SDK-supplied at pay time.** `execute_payment` takes only `{mandate_id, amount, expected_seq}`. The payee is read from the stored mandate on-chain, so a hostile SDK cannot redirect a payment to an attacker address.
- **Amount parsing is strict.** `toStroops` is ASCII-anchored and rejects negatives, scientific notation, hex, extra precision, and unicode / full-width / Arabic-Indic digits. Every adversarial input throws rather than producing a wrong integer.
- **No secret is logged, serialized, or put in an error message** anywhere in the signer or SDK path. Thrown errors carry only the mandate id and the contract's own message.
- **Package hygiene is clean.** Both packages publish `dist` only (`files: ["dist"]`), run no install or postinstall scripts, ship no secrets (verified with `npm pack --dry-run` and a secret-shaped grep over `dist`), and the `@reapp-sdk` scope is owned. Network constants (contract id, passphrase, native SAC) are correct, internally consistent across source and `dist`, and verified against on-chain reality.

## Fixed during this pass (`@reapp-sdk/core` 0.1.2)

Both were real but not exploitable. Each broke the strictness `toStroops` promises in its own docstring. Since the contract already rejects the dangerous outcomes, neither could move wrong funds against anyone. We fixed them so the SDK fails loudly on its own rather than leaning on the contract to catch a wrapped value.

1. **i128 amount bound.** `toStroops` produced an unbounded `BigInt`. A value past i128 max would silently two's-complement wrap at the ScVal encoding layer (the encoder does not range-check). `toStroops` now rejects any amount that does not fit i128.
2. **Safe-integer expiry bound (within u64).** `createIntentMandate` did not validate `expiry`. A `NaN`, fractional, or absurdly large value either threw cryptically at `BigInt()` or silently wrapped at the u64 encoder. `createIntentMandate` now requires `expiry` to be a positive integer of Unix seconds no greater than `Number.MAX_SAFE_INTEGER`. That stays comfortably within u64, so the value it hashes and sends is never lossy.

## Deferred to mainnet hardening (not testnet blockers)

- **Decimals source of truth.** `createIntentMandate` defaults to 7 decimals. A token with different decimals would scale the budget the user signs. Fetch the asset's `decimals` on-chain (or validate the supplied value) before mainnet. Bounded by the user-signed budget, never a bypass.
- **Allowance window vs mandate expiry.** `approveBudget` sets a fixed ~24h SEP-41 allowance window, decoupled from the mandate's own `expiry`. This fails closed (a lapsed allowance makes `transfer_from` revert), but for mainnet derive the allowance expiration from the mandate expiry.
- **Custom `NetworkConfig` is trust-critical.** Overriding `mandateRegistryId` reassigns where the allowance is granted. The default is the gate-checked testnet contract; this is self-inflicted config, but document it and consider a known-deployments guard for mainnet.
- **Exact dependency pinning and provenance.** Runtime deps use floating `^` ranges (`buffer` is already pinned). Pin exact versions and add publish provenance before mainnet.
- **Secret-handling ergonomics.** The API accepts a raw secret string for convenience. For mainnet, prefer a `Keypair` or external signer, drop the `keypair` field from the signer shape (it is unread by the SDK's signing path), and document memory-handling expectations.
- **Mandate-id canonicalization and nonce.** The id is a content hash over insertion-ordered JSON with a non-cryptographic nonce. This is not a security boundary (the contract authorizes by `require_auth(user)` and rejects duplicates with `AlreadyExists`), but adopt sorted-key canonicalization and a crypto nonce if other AP2 implementations ever recompute the id.

## Reproduce

The contract gate check and tests still hold (`cargo test`, 19/19). For the SDK, build the
workspace, run the live testnet end-to-end (8/8, no mocks, the same code published as
0.1.2), run the unit tests, and gate check any mandate straight from the chain:

```
npm run build
npm run e2e:sdk
npm test
npm run audit -- <mandate-id-hex>
```
